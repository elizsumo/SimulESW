SimulES Application Read Me

About SimulES: The main goal for a gamer in SimulES is to finish the project of software which is proposed when the game begins, this mean, the winner was the first player who implements and finishes his project without errors. This game allows player to learn about important computer science and software engineering concept. The resources into the game are: a game board, project cards, software engineer cards, game cards and one dice.

Motivation: The game technology has been used as a training tool in many educational areas but this technology isn’t common in the software engineering. SimulES is presented to minimize this problem; it is educational card game software which simulates the software development process. Therefore, this game allows students to take the role of a project manager and they manage related topics.

Game dynamic: When the game starts, one project must be chosen in the available projects set, this choice must be done randomly. Furthermore, the information about game and project would be visible to all players. After that, each player assembles in his individual board with his software engineer, problem cards, concept cards, and white/gray artifacts. The player who got the highest dice result was who chooses the project and starts the game. In each round, each player throws the dice once. The dice result allows player to buy a card, if the dice upshot is less than three, he will buy only cards (concepts and problems) according with this result. In contrast, if the dice upshot is more than three he will buy cards (concepts and problems) and software engineer cards, the quantity the software engineer cards will be the different between three and the result (dice upshot – 3). As an illustration, if Mary threw the dice and its result was 2, then Mary would buy 2 cards (problems and concepts). On the other hand, if Mary threw the dice and its result was 4 then Mary would buy 3 cards (problems and concepts) and 1 software engineer card. These cards will be to keep until that player believes necessary and makes his move into the game. Therefore, when the player put into the individual board some software engineer card this actions means that the player contracts this software engineer. Now, the player could receive problem cards of the others player. To be specific, the player named adversary throws cards problems to another player. The objective is to damage the other game and tries to win the game. These cards received will be used in the next round. After that, if the player has one card which invalidates some problem’s card, he will be able to use and the action describes in the problem card won’t affect his game, then he must discard both cards. On the other hand, if he doesn’t have any card which invalidates the problem’s card, this problem will be applied in his game. Furthermore, during the round the software engineer can execute one task set, his ability to execute these tasks is named “point of time”. These “point of time” allows he to execute different actions, such as:

Build artifact: the software engineer can build artifacts. If he builds with white cards then he will spend value of “point of time”. This value is indicated by complexity project. Or if he builds with gray cards then he will spend half value of “point of time”.

Inspect artifact: after building artifact task the software engineer could turn the cards (white or gray). It is possible some card would have defect.

Correct defect: if when the software engineer inspected some card he found some card with defect, he could correct this. But he spends “point of time”. In others words, He changes cards with defect by new cards with the same quality (white or gray).

Artifact integrate in the module: this task can do when the player has enough cards (white or gray) in his individual board and these can compose some requirement’s module by the project. This action removes into the individual board. In the same way, theses cards and they would be in other place out the individual board or may be in some place for integrating artifacts.

Once, the player has built all necessary components to finish the project then he can declare that he finished the game. After that one validating is done by the other players. That mean, some project modules would be randomly verified and they must be defects free. It is important that the number of modules has been verified depends on project quality. If the player doesn’t have any card with defect then he will winner the game.

Resources

Individual board: The individual board is an area where each player puts his software engineer in columns and his artifact in rows. The artifacts are the type of requirement, design, code, traceability and help. There is one cell next to the artifact and below the software engineer card where will be put the card (white and gray), as a result, this the software engineer build artifact with card (white and gray) in this place.

Project cards: These cards include the next information:

Description:text in natural languages which describes the main characteristics about the project. This description helps players understand the main requirements in the game. Moreover, this description could have references to articles and books related with the project or its main concepts.

Complexity: this indicates how many “points of time” the software engineer spends to complete one artifact. If the software engineer build with white cards this action costs half complexity project, in the contrasts, if the software engineer build with gray cards then this cards will cost half value of the white cards. For example if the complexity of project is 4 each white card cost 2 “point of time” and gray card cost 1 “point of time”. Size: this point indicates how many modules must be complete to integrate and finish the project. Six is the largest number of modules that would be built.

Quality: this point indicates how many defects free the final product must be. This value changes between one and five. This activity is done when the player deliveries his product.

Budget: quantity of points available to spend into the project. This budget will be one constraint when the player employs his software engineers. Moreover, during the game the sum of salaries related with software engineer who are contracted by player doesn’t exceed the budgets project.

Problem card: These cards describe classic problems about software engineering. These problems are result of failures into the development process. Furthermore, these cards allows player to create obstacles to others player.

The problem cards’ attributes are:

References: they help to know about characteristics problem.

Conditions: they are related with the game’s player

Effect: it is the impact into the game’s player

Concept card: These cards describe the good practices into the software engineering and they can be used by players. Therefore these cards allows player move to the proposed objectives.

Conditions: they are related with the game’s player

The cards’ attributes are:

Code and name

References: they help to know about characteristics and concept related.

Conditions: they are related with the player’s game

Effect: it is the impact into the player’s game

Cards (White and Gray): These cards represent the building product by the software engineer into the individual board. They could have bugs or could not. Furthermore they are white or gray, these colors represent their quality. To be specific, when a player uses a white card to build artifact then he spends twice “point of time”. On the contrary, if he uses a gray card to build artifact then he will spend a “point of time” to produces some artifact. Although he spends more “point of time” when he uses white cards, these cards don’t have much error probability than gray cards.

To win the game the player must build all modules through of the artifacts. These artifacts are defined when the game begins.
